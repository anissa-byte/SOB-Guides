AI Safety Policy

Overview & Core Rules

Every team member using AI tools at SOB is responsible for following these guidelines. This policy protects our clients, our team, and our business.

🎯 The Bottom Line
AI produces drafts. Humans make decisions. Nothing AI-generated reaches a client without a human reviewing and approving it first.
1
What's Approved for Use
▶
✓ Safe to use
  • Slash commands built by leadership
  • Skills in the skills/ folder of the team repo
  • Approved MCP connections (Airtable, Notion, Gmail, Slack, Google Calendar)
  • Claude Code inside Cursor for your daily work
  • Claude.ai for brainstorming and planning (separate from client work)
✗ Not approved
  • Random skills or plugins downloaded from the internet
  • Autonomous agents running without human oversight
  • Mac Mini or server-based deployments (not yet validated)
  • Any AI tool sending messages directly to clients without review
  • Third-party AI tools not vetted by leadership (ChatGPT, Jasper, etc. for client deliverables)
2
The Five Core Rules
▶
Rule 1 — Required

Never send AI output to a client without human review. Every output — DM copy, strategy recommendation, client message, email draft — must be reviewed and approved by a team member before it reaches a client.

Rule 2 — Required

Never put sensitive client information in AI prompts unless using approved tools. Client passwords, payment information, personal addresses, or health data should never be typed into any AI chat. Business data (MRR, funnel metrics, account names) is fine in approved tools like Claude Code with the Airtable MCP.

Rule 3 — Required

Never install unapproved skills, plugins, or MCP servers. If you find a tool that looks useful, use /feature-request to suggest it. Leadership will vet it before it enters the system.

Rule 4 — Required

Never share API keys. Not in Slack, not in email, not in a Notion doc, not in a commit. API keys are private credentials. If you think a key has been exposed, notify leadership immediately.

Rule 5 — Required

Never modify skill files directly. Skills are maintained by leadership. If a skill needs improvement, use /feature-request. Editing a skill file without approval can break it for the entire team.

AI Safety Policy

Client Data & Privacy

How to handle client information when using AI tools.

1
What You Can and Cannot Share with AI
▶
✓ OK to use in Claude Code
  • Client account names and brand names
  • Business metrics (MRR, conversion rates, funnel performance)
  • Brand voice descriptions and approved copy
  • Funnel strategies and content plans
  • Public social media content
✗ Never put in any AI prompt
  • Client passwords or login credentials
  • Payment information (credit cards, bank accounts)
  • Personal addresses, phone numbers, or health data
  • Social Security numbers or government IDs
  • Private DMs or personal conversations with clients
  • Any information the client has not shared with us for business purposes
2
How Client Data Flows Through Our System
▶
SourceWhat It ContainsHow AI Accesses It
AirtableAccount data, metrics, role assignmentsRead-only via MCP — pulled automatically when you run a command
Context LibraryVoice profiles, brand notes, funnel playbooksRead from files in the repo — never modified by AI
Approved Copy LibraryClient-approved hooks, CTAs, sequencesRead from Google Drive docs — CW and GS maintain manually
Key principle

AI reads client data through approved channels only. It does not scrape, search, or access client systems independently. Everything comes through Airtable MCP or files you've placed in the repo.

3
Data Retention & AI Memory
▶
  • Claude Code conversations are not permanently stored by Anthropic when using the API. Conversations end when the session ends.
  • Claude.ai conversations may be stored depending on account settings. Do not use Claude.ai for sensitive client data — use Claude Code in Cursor instead.
  • Files in the repo are version-controlled. Anything you commit to GitHub is tracked permanently. Be intentional about what you commit.
  • Start fresh conversations for new topics. Don't let client data from one conversation bleed into an unrelated task.
AI Safety Policy

Output Review & QA

Every AI output requires human judgment before it's acted on.

1
The Review Requirement
▶
Non-negotiable

No AI-generated content reaches a client without being reviewed by the responsible team member AND approved by leadership (Natasha or Anissa) during the testing phase.

This applies to:

  • DM funnel copy
  • Social media captions and story schedules
  • Email sequences
  • Client communication messages (Monday recaps, Slack updates)
  • Strategy recommendations
  • Build specs and automation documentation
2
What to Check in Every AI Output
▶
CheckWhat to Look For
AccuracyAre the numbers, client names, and facts correct? AI can hallucinate metrics.
ToneDoes it match the client's brand voice? Not generic AI-sounding language?
RelevanceIs it actually addressing what was asked? Does it fit the client's current stage?
SensitivityCould anything be misread, offensive, or inappropriate for this client's audience?
CompletenessIs anything missing that the client would expect?
HallucinationsDid the AI make up a product, stat, testimonial, or claim that doesn't exist?
⚠ Hallucination risk
AI can confidently state things that are completely false. Always verify specific numbers, dates, product names, and client-specific claims against real data. If it sounds too specific to be true, check it.
3
Batch Output Review Process
▶

When batch scripts run on a schedule, outputs go to a Notion review queue — not directly to clients. The review process:

  1. Batch runs and deposits outputs in the review queue
  2. The responsible team member reviews each output for their role
  3. Approved outputs are acted on. Rejected outputs get feedback via the feedback form.
  4. Feedback is routed to leadership to improve the skill
Best practice

Review batch outputs within 2 hours of them landing in the queue. Stale outputs may contain outdated data.

AI Safety Policy

API Keys & Credentials

API keys are private passwords. Treat them like your bank login.

1
Where Keys Should and Should Not Live
▶
✓ Safe locations
  • Your ~/.zshrc file (local to your machine, never committed)
  • The env section of your ~/.claude/mcp.json (local, never committed)
  • A password manager (1Password, LastPass, etc.)
✗ Never put keys here
  • Slack messages or channels
  • Notion pages or docs
  • Email
  • Any file inside the Git repo (it will be committed and visible to everyone)
  • Claude Code chat (it may echo it back in visible output)
  • Screenshots shared with others
2
If a Key Is Exposed
▶
Immediate action required

If you accidentally paste an API key into Slack, commit it to GitHub, or share it in any public/semi-public location:

  1. Tell leadership immediately — Slack DM to Natasha or Anissa
  2. Revoke the key — go to the provider's console and delete it
  3. Generate a new key and update your local config
  4. If it was committed to Git — do not try to fix it yourself. Tell leadership. The key is in the Git history even after deletion.
3
Key Rotation
▶
Recommended

API keys should be rotated (replaced with new ones) every 90 days or immediately after any team member departure. Leadership will coordinate rotation schedules.

AI Safety Policy

Git & Code Safety

Protecting the team repo and making sure changes are safe.

1
What Never Gets Committed
▶

The repo's .gitignore should already prevent most of these, but double-check before committing:

  • API keys or tokens — even in config files
  • .env files — these contain environment variables with secrets
  • node_modules/ — installed packages, not source code
  • Personal notes or drafts that aren't meant for the team
  • Client credentials or passwords
💡 Check before committing
Always review your staged changes in Source Control before hitting Commit. If you see a file you don't recognize or didn't intend to change, unstage it.
2
Branch Safety
▶
Required

Never commit directly to the main branch. Always create a personal branch (yourname-MM-DD-YY) and submit a pull request. Leadership reviews before merging.

  • Your branch is your sandbox — you can't break anything for other people
  • The PR process is the quality gate — changes only go live after review
  • If you accidentally make changes on main, tell leadership before pushing
3
AI-Generated File Changes
▶

When Claude Code edits files on your behalf:

  • Review the changes in Source Control before committing — make sure you understand what was changed
  • Don't blindly approve — Claude may edit files you didn't intend it to touch
  • If it modified a skill file — do not commit that change. Skill files are maintained by leadership only.
  • If you're unsure — ask Claude Code to explain what it changed and why
AI Safety Policy

Communication Boundaries

Where AI can and cannot send messages on your behalf.

1
Internal Communication (Team Channels)
▶
✓ Allowed
  • Claude Code sending messages to internal Slack channels (e.g., #ai-testing)
  • Claude Code creating Gmail drafts for your review
  • Claude Code creating or updating Notion pages in the team workspace
  • Claude Code creating calendar events
Best practice

Even for internal messages, review what Claude drafts before sending. AI can misunderstand context, get names wrong, or include information meant for a different conversation.

2
Client-Facing Communication
▶
Strict boundary

AI does not communicate with clients directly. Not now, not until leadership explicitly approves a supervised process. This includes:

  • No AI-sent emails to clients
  • No AI-sent Slack messages to client channels
  • No AI-generated Instagram DMs sent without approval
  • No AI-written client proposals sent without review

The workflow is: AI drafts → human reviews → human sends. Always.

3
Social Media & Public Content
▶
  • AI can draft social media captions, story schedules, and DM funnel copy
  • AI cannot publish anything directly to any platform
  • All public content goes through the standard approval flow: AI draft → CW review → GS confirm → client approval (if required) → publish

Incident Response

What to do when something goes wrong.

1
Types of Incidents
▶
SeverityExampleAction
LowAI output was wrong or low qualityFill out feedback form. Don't use the output. Continue working.
MediumAI edited a file it shouldn't have. A skill produced unexpected results for multiple clients.Don't commit the changes. Notify leadership in Slack. They'll investigate.
HighAPI key exposed publicly. AI sent a message to a client without approval. Client data appeared somewhere it shouldn't.Stop immediately. DM Natasha and Anissa. Do not try to fix it on your own.
CriticalClient credentials leaked. Unauthorized access to client accounts. Billing or payment data exposed.Stop everything. Call Natasha or Anissa directly. If unreachable, revoke the affected key/access immediately.
2
Reporting Steps
▶

For any medium-or-higher incident:

  1. Stop what you're doing with that tool or process
  2. Document what happened — screenshot or copy the relevant output
  3. Report via Slack DM to Natasha and Anissa with:
    • What you were doing
    • What went wrong
    • Which client(s) are affected
    • Any screenshots
  4. Wait for instructions before continuing with the affected tool or client
💡 No blame
Incidents are learning opportunities, not punishment. Reporting quickly is always better than trying to fix it quietly. The faster we know, the faster we can protect the client.
3
After an Incident
▶

Leadership will:

  1. Assess the impact and contain the issue
  2. Determine if the client needs to be notified
  3. Update the relevant skill, process, or policy to prevent recurrence
  4. Communicate the fix to the team

If a policy update comes out of an incident, it will be added to this page and announced in the team Slack.
